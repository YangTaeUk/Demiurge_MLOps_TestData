## 1. 데이터 제공 아키텍처

### 1.1 핵심 관점

인프라 자체가 서빙 레이어이다. 시딩이 완료되면 외부 시스템은 PostgreSQL, MongoDB, Kafka, MinIO에 **직접 접속**하여 데이터를 소비한다. REST/gRPC API는 보조 채널이다.

### 1.2 3계층 제공 모델

```
┌──────────────────────────────────────────────────────────────────┐
│                    외부 MLOps 시스템                              │
│  (Feature Store, Model Training, Data Validation, Monitoring)   │
└────────┬─────────────────┬──────────────────┬───────────────────┘
         │                 │                  │
    ┌────▼────┐      ┌────▼────┐       ┌────▼────┐
    │ Primary │      │Secondary│       │Tertiary │
    │ 직접접속 │      │ API     │       │온디맨드  │
    └────┬────┘      └────┬────┘       └────┬────┘
         │                 │                  │
    ┌────▼────┐      ┌────▼────┐       ┌────▼────┐
    │ PG:5434 │      │ REST    │       │Generator│
    │ Mongo   │      │ gRPC    │       │.fetch() │
    │ Kafka   │      │         │       │즉시 생성 │
    │ MinIO   │      │         │       │         │
    └─────────┘      └─────────┘       └─────────┘
```

### 1.3 계층별 특성 비교

| 계층 | 방식 | 장점 | 단점 | 적합한 시나리오 |
|------|------|------|------|---------------|
| **Primary** | 직접 인프라 접속 | 최대 성능, 네이티브 쿼리, 기존 도구 활용 | 네트워크 직접 접근 필요, 인프라 의존 | 데이터 엔지니어링, ML 학습 데이터 |
| **Secondary** | REST/gRPC API | 표준화된 인터페이스, 인프라 추상화 | 추가 레이턴시, 대량 데이터 전송 비효율 | 마이크로서비스 통합, 소량 조회 |
| **Tertiary** | 온디맨드 생성 | 인프라 불필요, 즉시 사용 | 시딩 데이터 미활용, 합성 한계 | 빠른 프로토타이핑, 단위 테스트 |

## 3. REST API 데이터 제공 (Secondary)

### 3.1 개요

시딩된 데이터를 HTTP API를 통해 제공한다. 외부 시스템이 인프라에 직접 접속할 수 없거나, 표준화된 인터페이스가 필요한 경우 사용한다.

### 3.2 기존 REST API와의 통합

기존 `api/rest/app.py`는 파이프라인 실행(`POST /pipeline/run`)과 제너레이터/어댑터 조회 엔드포인트를 제공한다. 데이터 조회 엔드포인트는 별도 라우터(`data_router`)로 구현하고, 기존 FastAPI 앱에 마운트한다:

```python
# api/rest/app.py — Phase 9에서 추가
from demiurge_testdata.api.rest.data_routes import data_router

app.include_router(data_router)  # /data/* 엔드포인트 자동 등록
```

### 3.3 신규 엔드포인트

기존 REST API (`api/rest/app.py`)에 데이터 조회 엔드포인트를 추가한다.

```
GET  /data/datasets                              # 시딩된 데이터셋 목록
GET  /data/datasets/{name}                       # 데이터셋 메타데이터
GET  /data/datasets/{name}/sample?limit=100      # 샘플 데이터 조회

GET  /data/{adapter}/query                       # 어댑터별 데이터 조회
     ?table=home_credit                          # RDBMS: 테이블명
     &where=target=1                             # RDBMS: 조건절
     &collection=instacart_orders                # NoSQL: 컬렉션명
     &filter={"order_dow": 0}                    # NoSQL: 필터 JSON
     &key=hm/transactions_train.parquet          # Storage: 객체 키
     &topic=ieee-fraud-events                    # Streaming: 토픽명
     &limit=100                                  # 최대 반환 건수
     &offset=0                                   # 페이지네이션 오프셋
     &format=json                                # 응답 포맷 (json|csv)
```

### 3.4 FetchPipeline 클래스

데이터 조회 흐름을 오케스트레이션하는 새 클래스:

```python
"""데이터 조회 파이프라인 — Adapter.fetch() → HandlerChain.decode()"""

from __future__ import annotations

from typing import Any, AsyncIterator

from demiurge_testdata.adapters.base import BaseAdapter
from demiurge_testdata.handlers.chain import HandlerChain


class FetchPipeline:
    """
    인프라에서 데이터를 조회하여 구조화된 형태로 반환한다.

    기존 push 파이프라인의 역방향:
    push: records → encode → push(bytes)
    fetch: fetch(query) → bytes → decode → records

    레코드 경계 보장:
        adapter.fetch()가 yield하는 각 chunk는 **완전한 디코딩 단위**이다.
        - RDBMS: 쿼리 결과 행 집합 (완전한 레코드 목록)
        - NoSQL: 문서 배치 (완전한 JSON 문서 목록)
        - Storage: 전체 파일 바이트 (단일 객체 read)
        - Streaming: 개별 메시지 (메시지 단위로 yield)

        따라서 chunk 경계에서 레코드가 잘리는 문제는 발생하지 않는다.
        각 chunk를 HandlerChain.decode()에 독립적으로 전달해도 안전하다.
    """

    def __init__(
        self,
        adapter: BaseAdapter,
        handler_chain: HandlerChain,
    ):
        self._adapter = adapter
        self._handler_chain = handler_chain

    async def fetch(
        self,
        query: dict[str, Any],
        limit: int | None = None,
    ) -> list[dict]:
        """
        어댑터에서 데이터를 조회하고 역직렬화한다.

        각 chunk는 adapter가 보장하는 완전한 디코딩 단위이므로,
        HandlerChain.decode(chunk)는 항상 정합성 있는 레코드를 반환한다.

        Args:
            query: 어댑터별 쿼리 딕셔너리 (03-어댑터-설계.md Section 11 참조)
            limit: 최대 반환 건수

        Returns:
            list[dict] — 역직렬화된 레코드 목록
        """
        results = []
        async for chunk in self._adapter.fetch(query, limit=limit):
            records = await self._handler_chain.decode(chunk)
            results.extend(records)
        return results

    async def fetch_stream(
        self,
        query: dict[str, Any],
        limit: int | None = None,
    ) -> AsyncIterator[dict]:
        """스트리밍 조회 — 대량 데이터에 적합"""
        async for chunk in self._adapter.fetch(query, limit=limit):
            records = await self._handler_chain.decode(chunk)
            for record in records:
                yield record
```

## 4. gRPC 데이터 제공 (Secondary)

### 4.1 Protobuf 메시지 정의

```protobuf
// protos/testdata/v1/data.proto

syntax = "proto3";
package testdata.v1;

// ── 서비스 정의 ────────────────────────────────
service DataService {
    // 단건 조회 — 쿼리 결과를 한 번에 반환
    rpc FetchData(FetchRequest) returns (FetchResponse);

    // 스트리밍 조회 — 대량 데이터를 청크 단위로 스트리밍
    rpc StreamData(StreamRequest) returns (stream DataChunk);

    // 데이터셋 목록 조회
    rpc ListDatasets(ListDatasetsRequest) returns (ListDatasetsResponse);
}

// ── 요청 메시지 ────────────────────────────────
message FetchRequest {
    string adapter = 1;         // "postgresql", "mongodb", ...
    map<string, string> query = 2;  // {"table": "home_credit", "where": "target=1"}
    int32 limit = 3;            // 최대 반환 건수
    int32 offset = 4;           // 페이지네이션 오프셋
    string format = 5;          // "json", "msgpack"
}

message StreamRequest {
    string adapter = 1;
    map<string, string> query = 2;
    int32 chunk_size = 3;       // 청크당 레코드 수 (기본 1000)
    string format = 4;
}

message ListDatasetsRequest {
    string category = 1;        // 필터: "relational", "document", ...
}

// ── 응답 메시지 ────────────────────────────────
message FetchResponse {
    string adapter = 1;
    int32 total = 2;
    bytes data = 3;             // 직렬화된 레코드 (format에 따라)
    string format = 4;
}

message DataChunk {
    int32 chunk_index = 1;
    int32 record_count = 2;
    bytes data = 3;
    bool is_last = 4;
}

message ListDatasetsResponse {
    repeated DatasetInfo datasets = 1;
}

message DatasetInfo {
    string name = 1;
    string category = 2;
    string seed_target = 3;
    int64 record_count = 4;
    bool is_seeded = 5;
}
```

### 4.2 gRPC 서버 구현

```python
# api/grpc/server.py (데이터 조회 확장)

class DataServiceServicer(data_pb2_grpc.DataServiceServicer):
    """gRPC 데이터 제공 서비스"""

    async def FetchData(self, request, context):
        """단건 조회: FetchPipeline → 직렬화 → 응답"""
        pipeline = self._create_fetch_pipeline(request.adapter, request.format)
        records = await pipeline.fetch(
            query=dict(request.query),
            limit=request.limit,
        )
        # 직렬화 후 반환
        encoded = await self._encode(records, request.format)
        return FetchResponse(
            adapter=request.adapter,
            total=len(records),
            data=encoded,
            format=request.format,
        )

    async def StreamData(self, request, context):
        """스트리밍 조회: 청크 단위로 yield"""
        pipeline = self._create_fetch_pipeline(request.adapter, request.format)
        chunk_size = request.chunk_size or 1000
        chunk_index = 0
        buffer = []

        async for record in pipeline.fetch_stream(query=dict(request.query)):
            buffer.append(record)
            if len(buffer) >= chunk_size:
                encoded = await self._encode(buffer, request.format)
                yield DataChunk(
                    chunk_index=chunk_index,
                    record_count=len(buffer),
                    data=encoded,
                    is_last=False,
                )
                buffer = []
                chunk_index += 1

        # 잔여 버퍼
        if buffer:
            encoded = await self._encode(buffer, request.format)
            yield DataChunk(
                chunk_index=chunk_index,
                record_count=len(buffer),
                data=encoded,
                is_last=True,
            )
```

## 5. 온디맨드 생성 모드

### 5.1 개요

인프라 시딩 없이 Generator가 즉시 데이터를 생성하여 반환한다. 기존 API 모드(`Generator.fetch(offset, limit)`)를 활용한다.

### 5.2 사용 시나리오

- 빠른 프로토타이핑: Docker 인프라 없이 데이터 확인
- 단위 테스트: 소량의 합성 데이터 즉시 생성
- CI 환경: 외부 인프라 접근 불가능한 환경

### 5.3 API 엔드포인트

```
GET /generate/{dataset}?offset=0&limit=100&format=json
```

```python
@data_router.get("/generate/{dataset}")
async def generate_on_demand(
    dataset: str,
    offset: int = Query(default=0, ge=0),
    limit: int = Query(default=100, le=10000),
    format: str = Query(default="json"),
) -> list[dict]:
    """
    온디맨드 생성: Generator.fetch(offset, limit) → 즉시 반환.
    인프라 시딩 없이 사용 가능.
    """
    generator = generator_registry.create(dataset)
    records = await generator.fetch(offset=offset, limit=limit)
    return [record.payload for record in records]
```

## 7. 인증 및 접근 제어

### 7.1 개발 환경 (기본)

Docker Compose 로컬 환경에서는 최소 인증만 적용한다:

| 인프라 | 인증 방식 | 자격증명 |
|--------|---------|---------|
| PostgreSQL | Password | testdata / testdata_dev |
| MongoDB | SCRAM-SHA-256 | testdata / testdata_dev |
| Kafka | 미적용 | - |
| MinIO | Access Key | testdata / testdata_dev_password |
| REST API | 미적용 | - |
| gRPC | 미적용 | - |

### 7.2 운영 환경 (확장 가능)

운영 환경에서는 다음 인증을 추가할 수 있다:

