## 1. 프로젝트 비전

### 왜 이 프레임워크가 필요한가

Demiurge MLOps 플랫폼은 14개 독립 모듈로 구성된 엔터프라이즈 MLOps 시스템이다. 각 모듈은 PostgreSQL(TimescaleDB), NATS, MinIO 등 다양한 인프라 컴포넌트에 의존하며, 실제 운영 환경에서 발생하는 대규모 데이터를 처리해야 한다.

그러나 개발·테스트 단계에서 **현실적인 데이터**로 인프라를 검증하기 어렵다:

- 운영 데이터를 개발 환경에 가져올 수 없다 (보안, 규정)
- 합성 데이터는 실제 분포·관계를 반영하지 못한다
- 인프라별로 데이터를 수동 변환·적재하는 작업이 반복된다
- 데이터 포맷과 압축 조합의 호환성을 체계적으로 검증할 수단이 없다

**Demiurge TestData**는 이 문제를 해결한다. Kaggle의 실제 데이터셋을 기반으로 다양한 인프라 시나리오를 자동으로 시뮬레이션하는 **데이터 파이프라인 프레임워크**이다.

### 핵심 가치

| 가치 | 설명 |
|------|------|
| **현실성** | Kaggle 데이터셋 32종으로 실제 분포·관계를 유지 |
| **범용성** | 22개 어댑터로 RDBMS, NoSQL, Streaming, Storage, FileTransfer를 모두 지원 |
| **조합 가능성** | 10종 포맷 × 7종 압축 = 70가지 변환 조합을 11개 클래스로 처리 |
| **자동화** | Generator → Handler → Adapter 파이프라인으로 수동 작업 제거 |
| **확장성** | Registry 패턴으로 새 컴포넌트를 코드 한 줄로 등록 |

## 2. 핵심 목표

### 2.1 인프라 한계 테스트

- 대량 배치 적재 (H&M: 3천만 트랜잭션)
- 실시간 스트리밍 부하 (IEEE Fraud: 이벤트 초당 수천 건)
- 고차원 IoT 데이터 처리 (Bosch: 4000+ 피처)
- 복잡한 관계형 스키마 조인 (Home Credit: 7개 테이블)

### 2.2 범용 데이터 커넥터

- **RDBMS 8종**: PostgreSQL, MySQL, MariaDB, MSSQL, Oracle, SQLite, CockroachDB, BigQuery
- **NoSQL 4종**: MongoDB, Elasticsearch, Redis, Cassandra
- **Streaming 5종**: Kafka, RabbitMQ, MQTT, Pulsar, NATS JetStream
- **Storage 3종**: S3/MinIO, Local Filesystem, HDFS
- **FileTransfer 2종**: FTP, SFTP

### 2.3 다중 모드 생성

| 모드 | 목적 | 사용 사례 |
|------|------|-----------|
| **Stream** | 실시간 시뮬레이션 | Kafka 이벤트 발행, IoT 센서 시뮬레이션 |
| **Batch** | 대량 일괄 적재 | PostgreSQL 초기 데이터 로드, S3 파일 업로드 |
| **API** | 페이지네이션 서빙 | gRPC/REST를 통한 데이터 요청-응답 |

## 3. 이해관계자와 사용 시나리오

### 3.1 주요 이해관계자

| 역할 | 관심사 |
|------|--------|
| **MLOps 엔지니어** | 파이프라인 안정성, 인프라 성능 검증 |
| **데이터 엔지니어** | 커넥터 호환성, 데이터 무결성 |
| **백엔드 개발자** | API 응답 속도, 동시성 처리 |
| **QA 엔지니어** | E2E 시나리오 재현, 데이터 정합성 |
| **인프라 운영** | 리소스 사용량, 장애 복구 검증 |

### 3.2 핵심 사용 시나리오

```
시나리오 1: 인프라 스트레스 테스트
  - H&M 데이터셋 → Parquet + LZ4 → Batch 모드 → PostgreSQL
  - 목표: 3천만 건 적재 시간, 메모리 사용량 측정

시나리오 2: 실시간 이상 탐지 시뮬레이션
  - IEEE Fraud → JSON + Snappy → Stream 모드 → Kafka
  - 목표: 초당 처리량, 지연시간 분포 확인

시나리오 3: 데이터 포맷 호환성 검증
  - Home Credit → 10종 포맷 × 6종 압축 → 각 어댑터
  - 목표: 모든 조합에서 encode/decode roundtrip 정합성 확인

시나리오 4: gRPC 데이터 서빙
  - Instacart → MessagePack + Zstd → API 모드 → gRPC 서버
  - 목표: 페이지네이션 성능, 동시 클라이언트 처리
```

## 4. 기술 스택 선정 근거

### 4.1 언어 & 런타임

| 기술 | 선정 근거 |
|------|-----------|
| **Python 3.11+** | Demiurge 플랫폼 생태계와 일치, 데이터 엔지니어링 라이브러리 풍부 |
| **asyncio** | 22개 어댑터의 I/O를 비동기로 처리하여 파이프라인 병목 최소화 |

### 4.2 핵심 라이브러리

| 라이브러리 | 역할 | 선정 근거 |
|-----------|------|-----------|
| **Pydantic v2** | 스키마 검증 | 120+ 컬럼 데이터셋의 타입 안전성, 직렬화 성능 (Rust 코어) |
| **pydantic_settings** | 환경 설정 | 부모 프로젝트와 동일한 설정 관리 패턴 |
| **PyYAML** | 설정 파일 | 사람이 읽을 수 있는 파이프라인 구성, YAML → Pydantic 검증 |
| **PyArrow** | 컬럼형 포맷 | Parquet, ORC, Arrow IPC 3종을 하나의 라이브러리로 처리 |
| **fastavro** | Avro 포맷 | 스키마 기반 직렬화, Confluent 생태계 호환 |
| **FastAPI** | REST API | 자동 OpenAPI 문서, Pydantic 모델 직접 활용 |
| **gRPC** | RPC API | 바이너리 프로토콜로 대량 데이터 전송 효율적 |
| **orjson** | JSON 직렬화 | Rust 코어 기반 6-10x 성능, datetime/numpy 네이티브 지원 |
| **cramjam** | 압축 통합 | 6종 압축 알고리즘 통합 (Gzip, Brotli, Snappy, LZ4, Zstd, LZMA), Rust 바인딩 |
| **msgspec** | MessagePack 직렬화 | 5-10x 성능, Struct 기반 제로카피 디코딩 |
| **FastStream** | 스트리밍 통합 | Kafka, RabbitMQ, NATS 3종 브로커 통합 async 프레임워크 |
| **fsspec** + **UPath** | 스토리지 통합 | S3, Local, HDFS, FTP, SFTP 통합 파일시스템 추상화 |

### 4.3 패키지 관리

| 도구 | 선정 근거 |
|------|-----------|
| **uv** | 부모 프로젝트와 동일, Rust 기반으로 pip 대비 10-100x 빠른 의존성 해결 |
| **pyproject.toml** | PEP 621 표준, optional dependency groups로 어댑터별 설치 분리 |

### 4.4 개발 도구

| 도구 | 용도 | 설정 |
|------|------|------|
| **ruff** >= 0.8 | 린팅 + 포매팅 | target: py311, line-length: 100 |
| **pytest** >= 8.0 | 테스트 | asyncio_mode = "auto" |
| **pytest-asyncio** >= 0.24 | 비동기 테스트 | 모든 어댑터·핸들러·제너레이터 테스트 |

## 5. 프로젝트 범위와 제약사항

### 5.1 범위 (In Scope)

- 32종 Kaggle 데이터셋 기반 데이터 생성 (6카테고리: Relational, Document, Event, IoT, Text, Geospatial)
- 22개 인프라 어댑터 (RDBMS 8 + NoSQL 4 + Streaming 5 + Storage 3 + FileTransfer 2)
- 10종 포맷 핸들러 + 6종 압축 핸들러
- 3가지 생성 모드 (Stream, Batch, API)
- Docker Compose 기반 로컬 인프라
- gRPC + REST API 서빙
- YAML 기반 파이프라인 설정
- 단위/통합/E2E 테스트

### 5.2 범위 밖 (Out of Scope)

- Kubernetes 오케스트레이션 (Docker Compose까지만)
- 운영 데이터 마이그레이션 도구
- 분산 처리 (Spark, Flink 통합)
- 실시간 모니터링 대시보드 (메트릭 수집까지만)
- 데이터셋 자동 갱신 (Kaggle API 다운로드 스크립트만 제공)

### 5.3 제약사항

| 제약 | 영향 | 대응 |
|------|------|------|
| Kaggle 데이터 라이선스 | 재배포 불가 | 다운로드 스크립트 제공, 원본 데이터 미포함 |
| 로컬 메모리 한계 | 대규모 데이터셋 전체 로드 불가 | Streaming 모드, 배치 크기 제한 (max 1M) |
| 어댑터 드라이버 호환성 | OS/플랫폼별 차이 | Optional dependency groups, CI 매트릭스 테스트 |
| 비동기 제약 | CPU-bound 작업 병목 가능 | ProcessPoolExecutor로 인코딩 오프로드 |
